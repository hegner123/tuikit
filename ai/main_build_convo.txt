  Ghostty's pageAllocator() hardcodes std.testing.allocator in test mode,
  regardless of what allocator we pass. The alternate screen buffer allocated by
   Ghostty uses this internal page allocator, which is the GPA. When nvim
  triggers alternate screen, Ghostty allocates via the GPA, and that allocation
  leaks because Ghostty's deinit path for the alternate screen might not be
  cleaning up the page pool properly in all cases.

  This is a Ghostty internal behavior ‚ÄĒ we can't fix it from our side. The
  pragmatic solution is to use std.testing.allocator for the session but accept
  the leak won't be caught by our code. Or better, just skip the nvim test in
  the test binary.
